## How to use this doc

- Each finding has a stable ID (`A1`, `A2`, …). IDs never change or get reused, even
  after a fix — so a commit can reference "fixes A1" forever.
- Update **Status** as work moves: `Open` → `In progress` → `Fixed` → `Verified`.
  Use `Won't fix` (with a reason) or `Deferred` where that's the call.
- When you fix one, add the commit hash in its detail section and a line in the Log.

**Status legend:** `Open` · `In progress` · `Fixed` (code changed) · `Verified`
(fix exercised against real data) · `Deferred` · `Won't fix`
**Confidence:** `CONFIRMED` (proven by reading code + all callers/data) ·
`PLAUSIBLE` (mechanism real, not exercised live)

## Detailed findings

### A1 — HIGH — CONFIRMED — Cancelling a tour booking never releases its seats
**Status:** Open
**Files:** `src/app/dashboard/bookings/actions.ts` (`changeBookingStatus`), `src/lib/queries/bookings.ts:172` (`updateBookingStatus`), `src/lib/scheduling.ts:39` (`claimTourSessionCapacity` release path — currently has zero callers)
Setting a tour booking to `cancelled` updates `bookings.status` only; `tour_sessions.booked_count`
is never decremented. The only seat-release path in the codebase is `releaseUnpaidBooking`
(webhook expiry + sweeper), guarded to `pending`/`awaiting_payment` — so a confirmed/paid/cancelled
booking keeps its seats counted forever. The camp path is fine (availability derives from
`status IN ('pending','confirmed')`); tours is the omission.
**Failure:** guest books 4, pays deposit, cancels; staff refund + set Cancelled → `booked_count`
stays +4 → departure shows full → real guests turned away. Also the reverse: `cancelled → confirmed`
is permitted, re-occupying inventory with no capacity guard (silent overbooking). **Very likely the
cause of the "availability looks wrong" reports.**
**Fix:** on a tour booking entering `cancelled` (and decide `no_show`) from `pending`/`confirmed`,
run a `db.batch()` mirroring `releaseUnpaidBooking` (guarded decrement + status in one transaction);
on transition *out* of `cancelled`, re-claim under the `booked_count + pax <= capacity - allotment_hold`
guard and refuse if full. Pairs with A16.

### A2 — HIGH — CONFIRMED — Camp price loop has no cap (DoS + overflow)
**Status:** Open
**Files:** `src/lib/pricing.ts:278` (`calculateCampPrice`), `src/app/[lang]/camp-booking-actions.ts:64` (`CampBookingSchema`), `:143` (`submitCampBooking`)
`calculateCampPrice` walks `while (cursor < end)` one day at a time. `submitCampBooking` validates only
`checkIn >= today` and `checkOut > checkIn`; the Zod schema puts no bound on the dates.
**Failure:** a direct POST with `checkOut=9999-12-31` runs ~2.9M iterations — **measured 538ms CPU,
subtotal ≈ ฿3.3 billion**. Burns the Worker CPU budget per request (cheap DoS via the only-IP-limited
`previewCampPrice`), and the absurd total flows into a real deposit if pushed through.
**Fix:** cap stay length — `.max()` bounds on the date fields plus a max-nights guard (e.g. reject
`checkOut - checkIn > 30` nights) in both `previewCampPrice` and `submitCampBooking`, and/or a hard
iteration cap inside `calculateCampPrice`. Tour path has no such loop — camp-only.

### A3 — HIGH — CONFIRMED — Sweeper uses current hold, not each session's frozen hold
**Status:** Open
**Files:** `src/lib/cron/expiry-sweeper.ts:69`, `src/lib/payments.ts:207`
Found independently by two reviewers. The sweeper cutoff is `now - CURRENT holdMinutes - 300s`, but each
Stripe session's `expires_at` was frozen at creation using the hold value *at that time*. The
"these can't drift" invariant only holds while the setting is constant — the admin Settings form can
move it (1439 → 30) anytime.
**Failure:** sessions created at hold=120min, admin lowers to 30min → next tick releases every
awaiting-payment booking older than ~35min while its Stripe page is live for up to ~85 more minutes →
guest pays → `markBookingPaid` no-ops → **paid for nothing** (compounded by A9's silent trail).
**Fix:** persist the deadline per booking — add a `payment_expires_at` column set from the same instant
`expires_at` is computed, and sweep on that column instead of re-reading the policy. **Needs a migration
+ a PGlite-style gate test** (per repo rule for schema changes).

### A4 — HIGH — CONFIRMED — Turnstile resets the wrong widget (3 on the landing page)
**Status:** Open
**Files:** `src/components/public/BookingWidget.tsx:187`, `CampBookingWidget.tsx:165`, `EnquiryForm.tsx:35`
All three carry a comment claiming "sole widget on its page" — false: `src/app/[lang]/page.tsx` renders
all three (Hero → BookingWidget, CampBookingSection → CampBookingWidget, EnquirySection → EnquiryForm),
so the landing page has **three** Turnstile widgets. A bare `reset()` resets only one, so for at least
two forms a failed submit resets a *different* form's widget and leaves the failing form's spent
single-use token in place.
**Failure:** one rejected submission (Zod error, rate limit, expired token) → every retry replays the
dead token → "couldn't verify you're human" forever until a full page reload. This is the exact class the
manage page already fixed.
**Fix:** give each form a ref on its `.cf-turnstile` div and call `reset(ref.current)`, same as
`ManageBookingRequestForm.tsx:69` / `WaiverForm.tsx:69`. Fix the three false comments.

### A5 — HIGH — CONFIRMED — Blank Settings number saves as 0
**Status:** Open
**Files:** `src/app/dashboard/settings/actions.ts:17` (`requiredNumber`), `src/app/dashboard/settings/page.tsx:85`
`requiredNumber` does `Number(String(raw ?? "").trim())` — `Number("")` is `0` (finite, passes). The
function is *named* requiredNumber but doesn't require. Inputs have `min` but no `required` (HTML skips
constraint validation on empty values).
**Failure:** admin clears "Daily AI budget" and saves → `dailyTokenCap` = 0 → read-side honors 0 as
"stop spending" → **chatbot goes dark, no error**. Same path: blank "Free cancellation window" saves 0
hours → guests lose the 72h free-cancel policy.
**Fix:** reject blank in `requiredNumber` (`if (!raw) throw`) and add `required` to all four number inputs.

### A7 — MED — CONFIRMED — "Today" is UTC, not Bangkok (+ no past-departure guard)
**Status:** Open
**Files:** `src/app/dashboard/day-sheet/page.tsx:5`, `src/components/public/BookingWidget.tsx:42`, `CampBookingWidget.tsx:37`, `src/lib/scheduling.ts:91` (server clamp), `booking-actions.ts:85`
`todayISO()` uses `new Date().toISOString()` (UTC) while the availability + camp calendars correctly shift
+7h. Bangkok is UTC+7, so 00:00–07:00 local shows *yesterday*.
**Failure:** the **day sheet shows yesterday's manifest between midnight and 7am Thailand time** — exactly
when crews prep ~07:30 pickups. Same UTC default lets the booking widget offer a departure dated
local-yesterday. Separately, there's **no `start_time` cutoff** — at 8pm a guest can still book today's
9am departure that already ran; the server trusts the client's `fromDate` (filters only `date >= ?`).
**Fix:** compute today in `Asia/Bangkok` on both surfaces, and add a server-side clamp
(`date >= Bangkok-today` + a same-day start-time / lead-time rule) in `listAvailableTourSessions` /
`submitTourBooking`.

### A8 — MED — CONFIRMED — Brevo send has no timeout
**Status:** Open
**Files:** `src/lib/brevo.ts:43` (`sendViaBrevo`), awaited inline by `booking-actions.ts:210`, `camp-booking-actions.ts:185`
Bare `fetch`, no `AbortSignal.timeout`. `sendBookingAck` (two sequential Brevo sends) is awaited before the
"Booked!" response returns — the exact failure mode `payments.ts` already bounds to 8s.
**Failure:** hung Brevo → guest waits indefinitely, concludes it failed, rebooks, double-claims a seat. Also
serially stalls the daily notification cron (40 guests × hang).
**Fix:** `signal: AbortSignal.timeout(10_000)` in `sendViaBrevo` (sibling of the `payments.ts` fix).

## Reviewed — no action needed

- **Rate-limit window edge** (`src/lib/rate-limit.ts`): fixed-window boundary can let up to 2× the limit through
  in a rolling window. Inherent to fixed windows, acceptable for this app. The `cf-connecting-ip` fallback is
  *not* spoofable (no `x-forwarded-for` used as the rate-limit key). Noted, not a bug.
- **Chat `MAX_TOOL_ROUNDS` overshoot** (`src/app/api/chat/route.ts`): a turn can make up to 5 model calls after
  the daily-cap check; bounded and documented ("overshoot by at most the in-flight turns"). Acceptable by design.

## Confirmed clean (high-confidence, worth knowing)

- **Stripe webhook**: signature verified over raw bytes; idempotent (claim-INSERT); a paid→refunded booking
  can't be walked back; partial refunds don't flip state.
- **Refunds**: admin-gated, server-side re-check, amount from Stripe's captured total (can't exceed paid),
  double-fire blocked.
- **Money math**: deposit + balance always equals total (balance by subtraction); blank money fields rejected in
  every admin parser *except* Settings (A5) and tour rates (A6); promo discount capped at subtotal (no negatives);
  satang conversion via `Math.round`.
- **Capacity invariant** `booked_count <= capacity - allotment_hold` used consistently — no bare-`capacity`
  comparison anywhere. Booking INSERT + capacity claim are one `db.batch()`.
- **Client-supplied prices**: never trusted anywhere (web, staff, chatbot confirm all recompute from D1);
  `tourId`/`zoneId` spoofs rejected against the authoritative session/unit row.
- **Chatbot booking mode**: confirm is a separate route the model can't call; draft claimed via guarded atomic
